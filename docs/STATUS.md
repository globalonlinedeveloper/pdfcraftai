When the fixture is a Node `Buffer` returned by `Buffer.from(base64, 'base64')`, that Buffer is backed by Node's shared slab allocator — `buf.byteOffset` can be nonzero, in which case the SOI read pulls two bytes of unrelated slab memory instead of the JPEG header. Node 22's slab layout happens to place tiny Buffers at offset 0 (sandbox passed); Node 18/20 runners don't (CI failed). Fix: switch `makeRedPng` and `makeBlueJpg` in `scripts/test-pdf-tools.mjs` from returning `Buffer.from(base64, 'base64')` to returning `Uint8Array.from(Buffer.from(base64, 'base64'))`. `Uint8Array.from(iterable)` iterates and copies bytes into a brand-new `ArrayBuffer`, so `byteOffset === 0` is guaranteed on every Node version. Verified locally (382/382 still green on Node 22) and via CI verification run `24709789261` (both `test (Node 18)` and `test (Node 20)` green). **Production safety**: grep'd for `embedJpg|embedPng` across the codebase — only `components/tools/ImageToPdfTool.tsx:146` uses them at runtime, and it already does `const bytes = new Uint8Array(await item.file.arrayBuffer())` before calling `embedJpg`. The File API gives a fresh byteOffset-0 ArrayBuffer, so production was never affected — the bug is isolated to the base64-decoded test fixtures. Docstrings added on both `makeRedPng` and `makeBlueJpg` explaining the bug so a future refactor that reaches for `Buffer.from` "because it's shorter" gets stopped at review. **This outcome validates the CI investment**: the workflow was shipped 15 minutes earlier as dev-hygiene infrastructure and paid for itself on its first run by catching a latent environment-dependent bug that had been silently sitting in the test harness.

## What got shipped (8 ships, all live)

### Visual editors — multi-page foundation

1. **#170 — Multi-page navigator** (`b004141`). `PageEditorTool` renders one page at a time with prev/next/page-input UI when `pageCount > 1`. New `lib/pdf/ops/rasterize-page.ts` for single-page render. 5 visual editors (Highlight, Redact, Add Hyperlinks, Sign, Free Draw) opted in.

2. **#171 — Per-page state persistence + multi-page apply** (`bd47500`). Discriminated union on `PageEditorToolProps<TState>` via `multiPage`. Apply receives `PageEditEntry<TState>[]` carrying pageIndex + state + dims snapshot. New `hasEdits` predicate decides which pages roll into entries; new `resetPageContent` predicate preserves global config (color, opacity, image) when navigating to a fresh page so users don't lose their tool config silently. "N other pages edited" pill in the navigator. Each consumer's apply iterates entries and chains bytes through one op call per page.

### Page-grid surface — DRY + perf

3. **#172 — `usePdfThumbnails` hook extraction** (`8b3517c`). Pulled near-identical 25-line blocks from PageGridTool / PdfSortPagesTool / PdfSplitTool into `components/tools/usePdfThumbnails.ts`. Hook owns thumbnails state, blob → object URL conversion, URL.createObjectURL/revokeObjectURL lifecycle, progress + rendering + error state. Returns `{thumbnails, rendering, progress, error, render(bytes), reset()}`. Net: -118/+179 LOC across 4 files; consumers shed ~75 LOC of duplication. Unmount cleanup uses a ref so revoke doesn't refire on every thumbnails change (subtle bug fixed in one place).

4. **#173 — Progressive thumbnail streaming** (`81664e2`). `lib/pdf/ops/rasterize.ts` gained `onPage(page: RasterPage)` callback firing per-page BEFORE `onProgress`; returning a Promise pauses the loop for back-pressure. `usePdfThumbnails.render()` pushes each thumbnail to state inside `onPage` as it arrives instead of awaiting the full batch. PageGridTool / PdfSplitTool relaxed grid-render gating from `stage === "ready"` → `thumbnails.length > 0 && stage !== "applying"`. For 500-page PDFs: page 1 lands at ~80ms instead of ~30-50s blank wait.

### Per-element editing UX

5. **#174 — Per-rect delete on Highlight + Redact** (`8324648`). X chip per rect, hidden during mid-drag and busy. `pointerdown stopPropagation` prevents chip click from firing the overlay's "start a new rect" handler. Redact chip is contrast-adaptive (dark on white cover, light on black cover).

6. **#175 — Per-stroke delete on Free Draw** (`b47e971`). Stroke-list panel in the config (sidesteps the canvas hit-testing ambiguity of overlapping strokes). Each row: color swatch + "Stroke N" + point count + width + X button. Index alignment preserved by mapping `state.strokes` (not `realStrokes`-filtered) and rendering null for sub-2-point in-flight strokes.

7. **#177 — Finger-friendly 40×40 tap targets on delete chips** (`a0221aa`). Wrapped visible 20×20 chip inside a 40×40 transparent button with chip centered via flex. Position adjusted from `top: -8` → `top: -18` so visible chip stays in same on-screen spot. Per Apple HIG / Material Design 44×44 minimums.

### Add Hyperlinks UX overhaul

9. **#178 — Output-PDF bugs fixed** (`9208d6c`). User reported the link annotation didn't work in Chrome's PDF viewer. Root causes: (a) `/Annots` array mutation didn't persist for pages whose annots reference is via indirect-ref chain (banking forms with AcroForm widgets hit this) — fix: always call `node.set(annotsName, annotsArr)` after mutation; (b) missing `/F 4` (Print bit, ISO 32000 §12.5.3) — Chrome's PDF viewer suppresses /Link annotations without it; (c) `/A` action dict was post-patched with PDFString URI — refactored to build action standalone with explicit PDFString. Plus preview-side fix: bumped saved-rect fill from 12% blue → 22% with 2px solid border + URL chip in top-left (12% was invisible against form-page backgrounds).

10. **#179 — Visible URL text stamped into output PDF** (`cec5f65`). Followup after user re-tested and reported "link not displaying". Root cause was deeper: `/Link` annotations are clickable-but-invisible by spec — every commercial PDF tool (Acrobat "Add Web Link", iLovePDF, Smallpdf) stamps visible URL text into the page content stream. Adopted same pattern: `drawText` URL inside each link rect as blue underlined Helvetica, sized to fit (`min(12, max(6, height × 0.65))`), ellipsis-truncated if too wide, plus 1pt underline below baseline. The /Link annotation still gets added on top so the entire rect is clickable.

11. **#180 — Drag saved rects to reposition** (`ad7e173`). Once saved, only delete-and-redraw could change a rect's position. Now `pointerDown` on a saved rect captures grab-point offset, `pointerMove` updates `rect.x/y` with bounds clamp, `pointerUp` commits. `useRef` for the move-loop state to avoid React reconciliation cost on every pointermove. `pointerdown stopPropagation` prevents the parent overlay's draw-new-rect from firing. Cursor `move` on hover, `grabbing` while dragging.

12. **#181 — Corner-resize handles** (`67787c8`). 4 handles (NW/NE/SW/SE), each a 24×24 transparent button with 10×10 visible blue-bordered white-fill indicator centered on the corner. Cursor mapping: NW+SE = nwse-resize, NE+SW = nesw-resize. Resize math snapshots ORIGINAL rect at pointerdown so all subsequent moves are absolute deltas (incremental drifts when clamping). Min-size 8×8 + page-bounds clamp + corner-anchor logic when shrinking past min so the rect doesn't drift sideways. Removed `overflow: hidden` from rect div so handles extending 12px past edge stay visible. Add Hyperlinks now matches Acrobat's "Add Web Link" UX surface.

### 42-tool standardization audit ships

13. **#182 — Visual progress bar + Rotate migrated to streaming hook** (`bc665aa`). Two findings from a deep audit of the 42 free client-side tools. (a) PageGridTool had `progress` state but rendered only numeric "47/500" — added a 4px gradient-filled bar with percentage chip + ARIA valuenow/valuemin/valuemax. Affects Extract, Delete, Sort, Split. (b) `PdfRotateTool` was the last tool still using raw `rasterizePdf()` — migrated to `usePdfThumbnails` for progressive paint (page 1 at ~80ms vs 30+ sec spinner). Net **−38 LOC** in Rotate plus consistent UX with the other thumbnail tools. Audit found Add Text Box multi-page wasn't actually a bug (op already loops every page by design) and Crop already had touch-action:none.

14. **#183 — Config panels show real current page, not "page 1"** (`f50124e`). Highlight + Redact config panels said "N highlights on page 1" even when user navigated to page 4 via the multi-page nav (active since #170). Stale label that lied after navigation. Fix: extended `PageEditorConfigProps` with `pageRender: PageRender` so config panels can read `pageIndex`. Non-breaking for existing config panels (just don't destructure if not needed). Runner always passes it; gated on `stage === "ready"` where render is non-null.

15. **#184 — Page-aware count labels in Free Draw + Add Hyperlinks** (`361920d`). Followup to #183. Free Draw said "N strokes · M points" and Add Hyperlinks said "N links saved" — neither lied but on multi-page PDFs users couldn't tell whether the count was per-page or summed. Now both labels include "on page X" when `pageCount > 1`; unchanged for single-page docs (terse for the common case). Both panels destructure the `pageRender` prop.

## Architectural patterns introduced

- **Slot-based generic runners** continue to scale: `<PageGridTool>` and `<PageEditorTool>` already proved the pattern; this session added a discriminated union (`multiPage: true | false`) on `PageEditorToolProps` to carry two apply signatures cleanly without breaking existing consumers (Crop, Add Text Box) that operate document-wide.
- **Hook-extracted shared state** for behaviors that span 3+ consumers: `usePdfThumbnails` owns the rasterize call + URL lifecycle and exposes `{thumbnails, render, reset}`. Pattern is reusable for any future "state derived from PDF bytes" feature.
- **Streaming through callback pumps**: `onPage` on the rasterize op feeds the hook, which feeds React state, which paints partial UI. Same pattern would work for any incremental computation.
- **Per-element delete via two routes**: in-canvas chips (Highlight, Redact) when elements are spatially distinct; sidebar list (Free Draw, Add Hyperlinks) when elements overlap or have non-spatial config. Both routes get `pointerdown stopPropagation` to coexist with the canvas's drag handlers.

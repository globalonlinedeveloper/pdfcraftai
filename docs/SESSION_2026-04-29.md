## Patterns established

1. **Idempotent Python wrap scripts** for mechanical refactors across many files.
   Examples: `wrap-set-error.py` (G1), `wrap-fetch-ai-retry.py` (M20),
   `wrap-uploaded-file-preview.py` (M18). Each script skips already-migrated files
   and warns on non-matching patterns, so they can be re-run safely.
2. **Static parse + assert** as a CI guard for cross-cutting invariants. Cheaper
   than runtime tests and catches the "developer forgot the new pattern" class
   of regression. See `scripts/test-*.mjs`.
3. **Hooks for one-shot mount effects** (`useHandoffConsumer`,
   `useFileUrlConsumer`, `useScrollErrorIntoView`) that consumers wire with one
   line. Each hook is single-purpose and composable.
4. **Module-level caches** (`lib/client/handoff.ts` registry, `useFirstPagePreview`
   LRU) backed by Maps with bounded size + LRU eviction. Survives Next.js
   client-side route changes; doesn't survive hard reload.

## Operational note — site 503 during high-velocity deploys

Around 17:30 UTC the site went 503 with the documented Hostinger LSAPI thread-cap
cascade pattern (see CLAUDE.md §5). ~25 commits in rapid succession triggered
auto-pull churn beyond what the cgroup thread budget could handle. SSH itself
became unreachable mid-handshake (the auth daemon was thread-starved too).

**Recovery procedure**:
1. Wait. The thread pool drains naturally — usually 5–15 minutes once new
   pushes stop landing.
2. If still 503 after 15 min: hPanel → Resource Usage → "Stop running process"
   manually. The app auto-restarts and the 503 clears.
3. **Do NOT** SSH-pkick repeatedly during a cascade — that adds load and
   prolongs the recovery. ONE pkick max per cycle (CLAUDE.md §5 wording).

The recent commits (post-`8453574`) are on `main` and will auto-pull once
the workers come back. No data risk; just availability.

### M24 — per-tool code splitting

`5327cb7`. Before this commit, `app/tool/[id]/page.tsx` statically
imported all 60+ tool runner components at the top of the file (51
import blocks across ~140 lines). Visiting any `/tool/{id}` route
shipped JS for every tool — even though users only run one. Webpack's
per-page splitting can't help when every component lives in the same
page file.

The fix split the dispatcher out of page.tsx (a server component)
into a new `components/tools/ToolRunner.tsx` ("use client" file) and
replaced static imports with `next/dynamic({ ssr: false })`. Each
dynamic() generates its own webpack chunk; visiting `/tool/merge`
now only loads PdfMergeTool's chunk plus its deps. Sibling tools
are not loaded.

Tools that share a base file (PdfTextExportTool / PdfRasterizeTool /
PdfChecklistTool / PdfSimpleOpsTool / SummarizeVariantTool /
StructuredVariantTool) each get separate `dynamic()` calls but share
the underlying chunk — webpack dedupes.

`ssr: false` because all tools are interactive client-side runners
that touch Blob, canvas, window.URL — none of which exist in the RSC
environment. The page already renders full longform/FAQ/metadata
around the tool for SEO; the tool's own JSX never had any business
being SSR'd.

Loading UX: a small "Loading tool…" placeholder during chunk fetch
so the area doesn't collapse. Empirically that gap is well under
500ms on a fresh tab over 4G, faster on warm cache.

Test guard: `scripts/test-tier1-expansion.mjs` Section B updated to
accept either the static `import { Foo } from "@/components/tools/Foo"`
shape OR the new dynamic `import("@/components/tools/Foo").then((m) =>
({ default: m.Foo }))` shape. The test now reads both page.tsx and
ToolRunner.tsx so wire-ups are pinned regardless of which file holds
them.

Build verification:
- `/tool/[id]` route shows **3.7 kB** page-specific JS (down from
  bundling all 60+ tool components into the page chunk).
- 116 webpack chunks generated for `client/`, one per dynamic-imported
  tool component plus shared infra.
- First Load JS shared by all: 89.1 kB.
- `npx tsc --noEmit`: clean.
- Tests: 3246 passed, 0 failed across 38 suites (unchanged).

### M12 — mobile keyboard occluding URL input

`62d3754`. PdfAddLinksTool's pending-link card sits below the page canvas;
on a small mobile screen, once the soft keyboard takes ~40% of the viewport
the URL input lands below the fold and the user types blind. Fix is one
onFocus handler that calls `scrollIntoView({block: "center"})` after a
280ms delay (matches iOS ~250ms and Android 200-300ms keyboard animation).
The delay matters — scrolling immediately lands on a position that's
wrong once the keyboard expands the visual viewport.

Scope: only this one input. Audit found PdfAddLinksTool is the only file
in `components/tools/` with `autoFocus`, and the only place where a
keyboard-occluded input is the actual UX failure mode reported. Other
text inputs (Stamp text, Add Text Box content, etc.) live in config
panels that scroll naturally with the page; their positions don't pin
them below the fold the way the pending-link card does after a
rectangle drag.

tool bundles) is the only remaining unshipped item — and that's
